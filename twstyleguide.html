Org Documentation Style Guide

This comprehensive guide establishes standards for voice, tone, terminology, and mechanics to ensure clear, consistent, and user-friendly documentation and UI text across all Org products.

How to Use This Guide:

Use the navigation panel on the left to quickly jump between major style categories.

1. Voice & Tone

Voice Principles: Be Helpful, Clear, and Confident

  • Clear and Concise: Use simple, direct language. Avoid unnecessary jargon or technical complexity.
  • Professional yet Friendly: Maintain a professional tone while being approachable. We are a guide, not a gatekeeper .
  • Active Voice: Prefer active voice over passive for directness. ( E.g., "You must save the file", not "The file must be saved by you".)
  • User-Centered: Write from the user's perspective using second person ("you") rather than first person ("we").

Tone Guidelines

Content TypeToneExample
Instructional ContentDirect, confident, and helpful"To begin, click the New Project button."
Error MessagesEmpathetic and solution-oriented"The connection failed. Check your network settings and try again."
Release NotesInformative and neutral"We've updated the API to include new endpoint parameters."
TroubleshootingPatient and supportive"If the issue persists, try clearing your browser cache."

2. UI Text Conventions

Capitalization Rules: Prioritize Sentence Case

CaseUse ForExample
Title Case Page titles, section headers, and main navigation menu items "Getting Started Guide", "User Management"
Sentence Case Button labels, form fields, and body text (Default) "Save changes", "Enter your email address"

Referring to UI Elements in Text

  • Bold the exact UI text: Click the Save button.
  • Use the same capitalization as appears in the UI.
  • Always include the element type for clarity (button, field, menu).
Action Verbs for UI: Use Click for a mouse action, Select for a choice from a list (e.g., radio button, dropdown), and Enter for typing in a field.
UI ElementHow to Write
ButtonsClick the [Button Name] button
LinksClick [Link Text]
CheckboxesSelect the [Checkbox Label] checkbox
Dropdown menusFrom the [Menu Name] dropdown, select...
Text fieldsIn the [Field Name] field, enter...
TabsNavigate to the [Tab Name] tab

3. Grammar & Punctuation

Serial Comma (Oxford Comma)

  • Always use the serial comma before "and" or "or" in lists to prevent ambiguity.
    • Example: The platform integrates with data, routing, and logistics systems.

Contractions

  • Avoid contractions in formal technical documentation for maximum clarity and ease of translation.
    • Example: don't, you're, we'll.
  • They are acceptable in informal UI text or blog posts.
    • Example: "You're all set!".

Numbers

  • Spell out numbers one through nine, use numerals for 10 and above.
  • Exception: Always use numerals for measurements, percentages, and technical values, regardless of size.
    • Example: 5 MB, 15%, 3 parameters.

Hyphenation (Compound Modifiers)

  • Use hyphens for compound adjectives that precede a noun. .
    • Example: "real-time data", "third-party integration".
  • Do not use a hyphen when the modifier follows the noun.
    • Example: "The data is in real time".

5. Standard Abbreviations

Rule: Always spell out on first mention in each major section: "Application Programming Interface (API)". After the first mention, use the abbreviation consistently. Do not introduce an abbreviation for a term used only once or twice.

AbbreviationFull FormWhen to Use
APIApplication Programming InterfaceUse freely after the first mention
BUBusiness UnitUse when describing organizational setups or team groupings.
CSVComma-Separated ValuesUse freely; no need to spell out.
SSOSingle Sign-OnUse after first mention in the section
UIUser InterfaceUse after first mention
URLUniform Resource LocatorUse freely; no need to spell out.

8. Feedback and Updates

This Style & Terminology Guide is a living document. Your feedback is crucial for its maintenance and accuracy. To suggest a change or addition:

  1. Create a Jira ticket in the Documentation project.
  2. Tag the ticket with the "style-guide" label.
  3. Provide a clear rationale and examples for the proposed change.
  4. Assign the ticket to the Technical Writer for review and approval.
